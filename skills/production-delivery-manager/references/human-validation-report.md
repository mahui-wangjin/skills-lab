# Human Validation Report

Use this reference to make production delivery reviewable by a human who does not want to inspect the full diff.

## Core Rule

Every medium, complex, or production-grade task must expose a Human Validation Packet in the final answer. The packet is the user's review surface.

Generate an HTML report when any report trigger applies. The report is the durable version of the packet plus evidence.

CI/workflow results are evidence sources, not the report itself. A workflow proves that a repeatable machine gate ran on a commit or branch. The report explains what that evidence proves, what it does not prove, and how it maps to architecture, core logic, browser behavior, and residual risk.

## Report Triggers

Generate an HTML report when one or more is true:

- the user asks for a report, evidence package, validation package, HTML report, or delivery trace
- the task is complex or production-grade and changes cross-module architecture, core business logic, permissions, security, data storage, schema, migration, queues, workers, external side effects, AI runtime, or release flow
- the task is a release candidate, merge-to-main, launch, rollback, or final acceptance task
- browser/UI validation uses screenshots, recorded traces, responsive checks, console/network inspection, or product acceptance evidence
- CI/workflow artifacts, local verification, screenshots, sub-agent findings, branches, or worktrees need a single acceptance view

Do not generate a report by default for tiny docs edits, typo fixes, one-line reversible changes, read-only explanations, or local scratch investigations unless the user asks.

## Default Location

Use this directory layout in the delivery target repository:

```text
.production-delivery-reports/
  YYYY-MM-DD_<short-slug>/
    index.html
    evidence/
      <screenshots-logs-traces>
    manifest.json
```

Rules:

- `index.html` is required when a report is generated.
- `evidence/` stores screenshots, log excerpts, exported traces, workflow artifacts, and other local artifacts referenced by the report.
- `manifest.json` records basic report metadata and evidence entries generated by the scaffold script.
- Use a stable, descriptive slug such as `2026-06-16_admin-permission-flow`.
- Prefer relative links from `index.html` to evidence files.
- If the target project already has a stronger report location, follow that policy and explain the deviation.
- If the report is part of merge/release evidence, keep it in the diff unless project policy says delivery reports are external artifacts.
- If the report is local-only, state that it was not committed and why.

## Scaffold Script

Use `scripts/create_report.py` to create the report directory, `index.html`, `evidence/`, and `manifest.json` whenever possible. This keeps report layout stable across agents.

Example:

```bash
python <skill-dir>/scripts/create_report.py \
  --title "Admin permission flow delivery" \
  --slug "admin-permission-flow" \
  --status candidate \
  --repository "E:/develop/code/my-project/example" \
  --delivery-target "master" \
  --output-root ".production-delivery-reports"
```

After scaffolding, edit `index.html` with task-specific evidence. Do not leave placeholder text in a final report.

## Evidence Grades

Classify evidence explicitly:

| Grade | Meaning | Examples |
| --- | --- | --- |
| `automated_assertion` | Machine-checked behavior | unit, integration, e2e, permission, migration diff tests, CI test jobs |
| `build_typecheck` | Compile/package safety | typecheck, build, lint, schema validate, CI build jobs |
| `browser_runtime` | Runtime UI evidence | console clean, network shape, route loads, Playwright/agent-browser path |
| `screenshot` | Visible state proof | desktop/mobile screenshots, before/after images |
| `manual_inspection` | Human/agent inspection | diff review, architecture review, docs comparison |
| `external_source` | Source-backed decision | official docs, issue references, ADRs |

Screenshots are valid evidence, but only for visible state. A screenshot can support "the UI rendered this state"; it cannot alone prove permissions, data isolation, idempotency, transaction safety, or backend correctness.

## CI And Workflow Evidence

When CI or a release-candidate workflow exists, include it in the evidence map:

- workflow name and run context: PR, push, branch, commit, or manual dispatch
- status: passed, failed, skipped, cancelled, or not run
- link or artifact path when available
- what it proves: repeatable gate, build safety, DB migration contract, browser e2e, release candidate hard gate
- what it does not prove: user acceptance, production deployment, real third-party integration, broad historical debt cleanup, or untested environments

If CI was not run, say so. Do not let local-only verification masquerade as CI evidence.

## Human Validation Packet

Use this packet in the final answer for medium and complex work. Keep it concise.

```md
### Human Validation Packet

- Status: <complete / candidate / partial / blocked / self-reviewed>
- Needs user decision: <no / yes: decision needed>
- Review first: <1-3 architecture or product decisions the user should inspect>
- Architecture surface: <main boundary, dependency direction, what should not have changed>
- Core logic surface: <critical flow and invariants>
- Evidence map: <tests/build/CI/browser/screenshots/docs and what each proves>
- Evidence gap: <what was not proven>
- Steelman result: <strongest objection and decision>
- Residual risk: <specific risk or none known beyond untested environments>
- Human checkpoints: <2-5 checkboxes/questions for acceptance>
- Report: <path to index.html or "not generated: reason">
```

For small work, compress the same facts into one paragraph.

## HTML Report Contract

The HTML report must be readable offline and reviewable without build tools.

Requirements:

- no external network assets, CDNs, fonts, analytics, or remote scripts
- no secrets, tokens, session values, unredacted PII, or full noisy logs
- include generation date, task title, repository/workspace, branch or worktree, and delivery target
- put the validation verdict near the top
- separate facts from assumptions and residual risks
- link screenshots, logs, traces, and CI artifacts with captions and evidence grades
- preserve failed, skipped, or unrun checks; do not hide them in appendices
- include steelman objections and the final decision for each material objection
- include human checkpoints that can be accepted or rejected without reading code

Use a quiet, utilitarian layout. Avoid decorative styling that weakens scanning.

## Required Sections

1. **Executive Validation**
   - status, delivery target, decision needed, report scope
2. **Human Checkpoints**
   - architecture, core logic, user-visible behavior, residual risk acceptance
3. **Architecture Review Surface**
   - changed boundary, dependency direction, module ownership, non-goals
4. **Core Logic Review Surface**
   - critical flow, invariants, error paths, permission/tenant/audit/idempotency notes when relevant
5. **Evidence Map**
   - commands, tests, CI/workflow results, browser checks, screenshots, source documents, what each proves, and limitations
6. **Browser Evidence** when UI/browser work occurred
   - desktop/mobile screenshots, route, viewport, action path, console/network result, limitations
7. **Steelman Counter-Review**
   - strongest objections, evidence checked, decision, remaining risk
8. **Workspace and Integration**
   - original workspace, branch/worktree, delivery target, integration status, cleanup status, unrelated changes
9. **Appendix**
   - concise file list, important diffs by intent, skipped checks, follow-up tasks

## Evidence Hygiene

- Redact secrets and personal data before saving screenshots or logs.
- Prefer short log excerpts over full command output.
- Link large traces instead of embedding them.
- If evidence is generated by a browser tool, include route, viewport, user action path, console result, and network/API result when relevant.
- If a screenshot is only illustrative, label it as `screenshot` and state what it cannot prove.
- If a CI artifact is external and cannot be bundled, include a link or run identifier and state whether the report remains useful offline.

## Final Answer Contract

When a report is generated, final handoff must include:

- report path
- whether it is committed, staged, or local-only
- evidence directory path
- CI/workflow run status if relevant
- any screenshots, logs, or workflow artifacts omitted and why
- one sentence explaining what the user should inspect first

Do not claim full production completion if the report is missing a required evidence surface for the risky part of the task.
