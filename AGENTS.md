# AGENTS.md

## 项目规则

### Memory MCP Project Rules

- 项目标识：`skills-lab`
- 仅当本轮任务直接涉及本仓库的业务、架构、流程、设计、文档或代码实现时，才读取和写入项目 memory。
- 新会话开始时，若为项目任务，先读取 `skills-lab` 的最近观察，并输出 3 条当前状态与 3 条下一步。
- 会话结束时，若为项目任务，写回：目标、关键改动、风险/阻塞、下一步。
- 若本轮是全局工具链、环境配置或通用经验，不写入项目 memory。
- 只有对下次继续做这个项目仍有帮助的长期事实，才写入项目 memory。
- 形成长期项目事实时同步更新 `docs/`；文档变更时同步更新 `docs/README.md`。

### Skill Enforcement Rules

- 当任务涉及新增依赖、新服务、新基础设施、新自研模块、重大重构时，必须先使用 `reuse-first-guard`。
- 当任务明显在搭建高复用通用能力时，默认也必须先使用 `reuse-first-guard`；典型范围包括鉴权、支付、搜索、存储、消息、表单、上传、CMS、监控、工作流。
- 若 `reuse-first-guard` 判断存在更低成本或更低维护代价的复用路径，默认先给出替代方案与推荐结论，再决定是否进入自研实现。
- 若用户仍要求自研，必须明确说明业务壁垒、合规、安全、性能、供应商锁定或 TCO 等理由，再继续实现。
- 使用 `reuse-first-guard` 时，当前工程已有实现、模板、示例、测试、适配器和 demo 是第一复用来源；除非用户明确要求新写，或已证明本地模式不适配，否则默认先复制并适配最近的本地实现。
- 对 Dialog、Select、Combobox、DatePicker、Upload 这类通用交互，默认采用“UI 可自绘，行为层优先原生/官方/Headless/成熟领域库”的分层决策，不默认自研状态机。
