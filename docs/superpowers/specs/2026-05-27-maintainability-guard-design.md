# maintainability-guard 设计说明

## 背景

长期 AI 协作开发中，前端页面、后端服务、worker、脚本和测试辅助代码都容易在连续补功能时变成巨型文件。典型表现是 Agent 每次为了完成当前缺口继续在同一个文件里打补丁，短期可运行，长期不可维护。

本仓库新增 `maintainability-guard`，用于在开发前识别是否允许继续原地开发，还是必须先做边界澄清或小步拆分。

## 设计原则

- 行数是报警器，不是唯一裁判。
- 不为了小文件洁癖拆分。
- 真正拆分理由是职责混杂、关注点缠绕、变化轴分离、复用出现、测试困难、依赖方向混乱或接口不稳定。
- 示例可以引用具体框架，但通用判断必须保持方法论层级。
- skill 负责 Agent 行为与判断，自动化脚本和 lint 负责客观门禁。

## 核心判断

`maintainability-guard` 将以下原则设为一级检查项：

- 职责分离：一个模块只负责一个主要变化原因。
- 关注点分离：业务规则、接口适配、持久化、权限、审计、展示转换不要长期混在一起。
- 高内聚低耦合：相关逻辑放一起，不相关逻辑不要互相知道太多。
- 单一抽象层级：一个函数里不要既写业务决策，又写 SQL/DOM/HTTP 细节。
- 变化轴识别：经常一起变化的放一起，经常独立变化的拆开。
- 测试边界：无法单独测试的逻辑，通常边界不清。
- 依赖方向：领域规则不要依赖外部框架细节。
- 稳定接口：模块之间通过清晰契约协作，而不是互相伸手拿内部状态。

## 阈值口径

- `< 400` 行：正常开发，除非职责已经混杂。
- `400-800` 行：观察区。
- `800-1200` 行：新增行为前必须输出 Maintainability Gate。
- `1200-2000` 行：必须给出不拆理由，或先拆一个明确职责。
- `> 2000` 行：默认禁止继续新增业务行为。
- `> 3000` 行：视为结构恢复事件，先做恢复计划。

## 后续增强

后续可补 `scripts/check-code-maintainability.ps1` 或各项目自行接入 ESLint、dependency-cruiser、jscpd、knip、Semgrep、SonarQube 等工具，把部分阈值和依赖方向规则转为机器检查。
