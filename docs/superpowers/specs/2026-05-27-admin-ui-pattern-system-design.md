# admin-ui-pattern-system 设计说明

## 背景

后台管理界面开发容易出现“模板组件堆叠”问题：Agent 看到接口字段、表格、表单、卡片和图谱组件后，直接把多个列表、说明、创建表单、详情和操作按钮堆在一个页面中，导致页面虽然有组件但不可用。

现有通用 UI skill 多数关注视觉、组件质量或具体框架能力；具体后台平台 skill 又常绑定某个框架或产品。`admin-ui-pattern-system` 需要解决的是更稳定的方法论问题：在任意后台模板或管理系统框架中，先盘点当前模板能力和完整页面示例，再归纳项目级后台页面范式，确认后才编码。

## 目标

- 让 Agent 在后台页面开发前先做业务任务建模，而不是从 API 字段直接生成页面。
- 让 Agent 先盘点当前模板/框架已有能力，包括搜索折叠、表格工具栏、列设置、刷新、全屏、抽屉、弹窗、详情页、树表、Tabs、状态组件和完整示例。
- 在模板没有明确标准范式时，要求 Agent 从示例组件和完整页面中归纳候选项目范式，并与用户确认。
- 固化字段多、搜索多、操作多、状态多时的体验治理规则。
- 固化承载优先级：主页面只放主任务；详情、编辑和次级信息默认优先抽屉；信息过多或流程复杂时使用独立页面；短确认和极轻量输入才用弹窗。
- 为图谱/画布、审批/流程、配置治理、审计观测等后台复杂页面提供通用范式参考。
- 补齐非 CRUD 页面范式，覆盖总览工作台、分步向导、导入映射、规则构建器、运行监控、Diff/发布确认、权限矩阵、对象 360、异常复核台、消息运营、搜索选择器、时间线、日历排期、报表分析和模板编辑。
- 增加完成门禁和降级规则：缺少模板证据、线稿确认、状态覆盖或浏览器验收时，不得声称产品级完成。

## 非目标

- 不绑定 Vben、Ant Design、Element Plus、MUI、React Admin、shadcn 或任何特定框架。
- 不重新定义底层视觉设计系统、颜色、字体、间距或组件 API。
- 不要求所有项目使用同一套具体组件，而是要求先发现并复用当前项目已有模板能力。
- 不替代真实浏览器验收；重要后台流程仍需要截图或浏览器自动化证据。

## 核心规则

1. 后台页面不能从 API 字段、数据库表或组件清单直接堆出来。
2. 编码前必须输出页面类型、主用户、核心任务、主对象、次对象/关系对象、模板能力盘点、参考示例、项目范式建议、搜索策略、字段分级、操作收纳和状态覆盖。
3. 字段超过 6 个必须分 P0/P1/P2/P3；搜索条件超过 4 个必须折叠；行操作超过 3 个必须收纳。
4. 详情、编辑、审计、字段说明、权限决策、运行证据和上下文预览默认优先使用抽屉。
5. 如果弹窗需要滚动、分组、Tabs、长表单或大量说明，默认升级为抽屉或独立页面。
6. 图谱/画布页面必须画布优先，图例、帮助、属性编辑和新增关系流程不得长期挤压主画布。
7. 非 CRUD 页面必须先选择主范式；复杂流程不能强行套普通 CRUD 表格。
8. 缺少模板/示例证据、线稿确认、状态覆盖或浏览器验收时，只能交付候选方案、局部修复或自审结果。

## 验证方式

- 运行官方 `quick_validate.py` 校验 skill。
- Windows 中文环境下运行校验前设置 `PYTHONUTF8=1`。
- 运行 `npx skills add . --list --full-depth` 确认仓库可发现该 skill。
- 对典型 smoke prompt 验证触发方向：
  - `我要新增一个后台管理页面`
  - `这个后台页面字段很多、操作很多`
  - `图谱编辑器右侧表单太挤`
  - `我要做一个规则配置向导`
  - `我要做数据导入字段映射页`
  - `我要做权限矩阵和最终权限预览`
