# reuse-first-guard Design

## 背景

当前仓库 `skills-lab` 主要沉淀可安装、可复用的开发类 skill。现有核心 skill `design-to-frontend-delivery` 解决的是前端设计落地问题，但仓库缺少一个面向全局开发决策的通用 skill，用于在关键工程决策前先判断“是否应优先复用现成能力，而不是直接自研”。

目标不是做一个泛泛的“技术选型建议器”，而是做一个能在关键决策点形成稳定约束的 skill：

- 日常开发中贯彻“能不自己写就不自己写”的原则
- 在新增依赖、服务、基础设施或新自研模块时，先检查复用路径
- 当涉及技术路线时，输出简洁但可执行的选型判断
- 不把所有开发变成审批流，而是在关键决策点做半阻断

## 目标

创建一个名为 `reuse-first-guard` 的全局通用 skill，供用户安装后在各类开发项目中复用。该 skill 的目标是：

1. 在关键开发决策前，强制执行“复用优先”检查。
2. 在存在更低成本、更低维护代价的现成方案时，默认先推荐复用路径，而不是直接进入自研。
3. 在确实需要自研时，要求明确说明自研理由与约束来源。
4. 在关键路线选择场景下，输出一份短格式、可执行的技术决策结论。

## 非目标

以下内容不属于本 skill 的职责范围：

- 不负责替用户完成完整的市场调研报告
- 不负责细化到具体商业合同或采购审批流程
- 不替代后续领域型实现 skill，例如前端交付、测试、部署
- 不在普通小改动、低风险代码编辑中强制介入

## 适用场景

该 skill 适用于以下场景：

- 新增依赖
- 新增第三方服务
- 新增基础设施或平台能力
- 新增自研模块
- 重大重构
- 明显在重复造轮子
- 明显在搭建通用能力，例如鉴权、支付、搜索、存储、消息、CMS、表单、埋点、监控、工作流

该 skill 不要求在每次普通实现前都触发，只在命中上述关键决策场景时强制触发。

## 核心原则

该 skill 贯彻的核心原则为：

`先复用，再采购/集成/生成，最后才自研。`

这里的“复用”是广义概念，包括：

- 官方能力
- 托管服务或 SaaS
- 成熟开源库、模板、starter
- 代码生成或 AI 生成

因此该 skill 虽命名为 `reuse-first-guard`，但其判断范围并不局限于代码复用，而是覆盖“是否有更低成本现成路径可用”的整体决策。

## 执行模型

### 阶段 1：复用优先检查

在命中强制触发条件后，优先执行复用优先检查。检查顺序固定为：

1. 官方能力
2. 托管服务 / SaaS
3. 成熟开源库 / 模板 / starter
4. 代码生成 / AI 生成
5. 自研

该阶段的目标不是列举所有可能性，而是快速判断是否存在明显更优的非自研路径。

若存在合理复用路径，则默认进入半阻断输出，不直接开始自研实现。

### 阶段 2：技术选型评估

当满足以下任一条件时，进入技术选型评估：

- 阶段 1 无法直接收敛到明确推荐项
- 多个复用路径都可行，但取舍差异明显
- 任务本身已明确涉及技术路线选择
- 用户明确要求做技术选型判断

该阶段输出简版决策结论，而不是长报告。至少包括：

- 推荐方案
- 备选方案
- 主要权衡
- 弃选理由
- 是否允许自研

## 半阻断策略

本 skill 采用“半阻断”而非“强阻断”。

规则如下：

1. 若 skill 判断“不建议直接自研”，则默认先给出替代方案和推荐结论。
2. 在给出替代路径前，不应直接进入实现。
3. 若用户仍坚持自研，可以继续，但必须明确记录自研理由。

允许继续自研的理由限定为以下类别：

- 业务核心壁垒
- 合规或数据主权要求
- 安全要求
- 性能要求
- 供应商锁定不可接受
- 长期总拥有成本（TCO）明显优于复用方案

如果无法给出上述任一类理由，则默认不应建议自研。

## 输出格式

该 skill 使用固定短格式输出，避免变成冗长咨询报告。

```md
### Reuse-First Check

- 触发原因：<为什么命中>
- 可复用路径：<官方 / SaaS / 开源 / 生成>
- 推荐方案：<推荐项>
- 不建议直接自研的原因：<1-3 条>

### Decision

- 是否允许直接自研：<否 / 有条件允许 / 是>
- 若继续自研，必须满足：<理由或约束>
- 下一步建议：<安装某库 / 选某服务 / 做一个 POC / 再补评估>
```

该格式的设计目标是：

- 足够短，适合嵌入开发会话
- 足够硬，能形成行为约束
- 足够结构化，便于后续扩展评估维度

## Skill 结构设计

建议在仓库中按以下结构创建 skill：

```text
skills/
  reuse-first-guard/
    SKILL.md
    agents/
      openai.yaml
    references/
      evaluation-criteria.md
```

各文件职责如下：

- `SKILL.md`
  - 只保留核心触发规则、两段式流程、半阻断要求、输出格式
- `references/evaluation-criteria.md`
  - 存放评估维度、允许自研的例外条件、常见判断准则
- `agents/openai.yaml`
  - 提供 UI 展示元数据，便于安装后被理解和发现

## 仓库级同步要求

创建该 skill 时需要同步以下内容：

1. 更新根 `README.md`
   - 将 `reuse-first-guard` 加入 `Available Skills`
   - 补安装命令与简短说明
2. 更新 `docs/README.md`
   - 将该 skill 及其设计文档纳入索引
3. 更新项目规则
   - 在项目规则中加入“命中关键决策场景时必须先使用该 skill”的硬性约束

## 项目规则接入方式

建议在项目规则中增加如下硬性规则：

- 当任务涉及新增依赖、新服务、新基础设施、新自研模块、重大重构时，必须先使用 `reuse-first-guard`
- 若 skill 判断存在更低成本复用路径，默认先给出替代方案与推荐结论
- 若仍要自研，必须明确说明自研理由，再进入实现

这条规则的作用是：

- 规则层负责约束“何时必须触发”
- skill 层负责约束“触发后如何判断”

## 测试与验收

该 skill 完成后，至少应进行以下验证：

1. `我要做一个用户鉴权模块`
   - 应优先推荐官方能力、托管服务或成熟方案，而不是直接自研
2. `我要给现有系统接入支付`
   - 应优先走第三方支付方案，并说明不建议从零实现支付流程
3. `我要做一个只属于我们业务的核心排程引擎`
   - 可在说明业务壁垒、性能或 TCO 理由后，给出“有条件允许自研”
4. `我要换掉项目里的表单库`
   - 应判断是否属于关键依赖调整，并输出简版选型建议

验收标准：

- 命中关键决策时能触发
- 能在“现成方案优于自研”时形成半阻断
- 能在确需自研时给出清晰的例外条件
- 输出结构稳定、短小、可执行

## 风险与边界

主要风险如下：

- 若触发范围写得过宽，skill 可能在普通开发中频繁打扰
- 若触发范围写得过窄，skill 可能失去约束力
- 若输出过长，会降低在真实开发会话中的使用率
- 若“允许自研”的例外条件定义不清，skill 会沦为形式化提醒

为降低这些风险，首版应坚持：

- 只在关键决策场景强制触发
- 输出短格式
- 例外条件有限且明确

## 实施顺序

1. 创建 `reuse-first-guard` skill 目录与模板
2. 编写 `SKILL.md`
3. 编写 `references/evaluation-criteria.md`
4. 生成并校对 `agents/openai.yaml`
5. 更新根 `README.md`
6. 更新 `docs/README.md`
7. 更新项目规则中的硬性约束
8. 运行 `quick_validate.py`
9. 做最小 smoke check
