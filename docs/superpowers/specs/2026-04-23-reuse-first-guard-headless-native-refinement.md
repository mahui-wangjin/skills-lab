# reuse-first-guard Headless/Native Refinement

## 背景

在真实项目 `metlife-huodong` 的上传控件迭代中，已经先触发了 `reuse-first-guard`，但后续实现仍然从“最小 demo 壳层”滑向了自研状态逻辑，出现了典型的通用控件 bug：

- 同名文件重复进入列表
- 删除错位
- 上限穿透
- 桌面端结构调整时，上传区和提交区的状态与布局耦合扩大

这说明旧版 `reuse-first-guard` 虽然能阻止直接上大而全的自研方案，但还没有把一个关键规则说死：

`自定义 UI` 不等于 `自定义行为层`。

## 本轮设计目标

为 `reuse-first-guard` 增加一个更稳定的中间判断层：

1. 当用户要求“样式自己做”“界面自己实现”时，默认只把它理解为视觉层授权。
2. 对表单、选择器、弹层、日期、上传等通用交互，优先复用：
   - 浏览器原生能力
   - 框架官方能力
   - Headless / unstyled primitives
   - 成熟领域库
3. 在没有真实后端契约时，禁止把 demo 级文件选择继续扩成自研上传状态机。

## 新增决策规则

### 规则 1：视觉层与行为层拆分

- 视觉层：布局、样式、动效、品牌表达，可自定义。
- 行为层：状态转换、校验、焦点管理、键盘行为、文件选择、上传队列、错误恢复，默认不自研。

### 规则 2：上传场景的特殊限制

若用户当前只是要一个“上传控件 UI”或“上传占位体验”，但：

- 没有真实上传接口
- 没有失败重试
- 没有进度和异步生命周期定义

则只允许：

- 原生 `input[type=file]`
- 最薄的本地展示壳层

默认不允许继续扩到：

- 去重引擎
- 队列管理
- 重试机制
- 复杂校验矩阵
- 自研上传状态机

### 规则 3：成熟行为层优先

对于通用交互控件，推荐顺序应细化为：

1. 浏览器原生能力
2. 框架官方能力
3. Headless / unstyled primitives
4. 成熟领域库
5. 薄适配层
6. 自研

## 预期效果

迭代后的 `reuse-first-guard` 在未来应更稳定地输出这类结论：

- “你可以继续自定义 UI，但不要默认自研通用行为层。”
- “当前更合理的是原生 file input + 自定义视觉壳层，而不是继续手写上传状态机。”
- “如果真实上传能力要落地，再进入成熟上传库或平台集成的选型，不要在 demo 逻辑上继续生长。”
